# 视觉检测软件 AI 训练与接入规划

## 1. 目标说明

本文档用于整理当前视觉检测项目引入 AI 训练能力时的总体思路、核心原理、实施流程与分阶段规划，作为后续方案评审、数据准备、模型训练、软件集成与现场上线的基础说明。

当前项目已经具备以下基础能力：

- 配方管理
- 模板与 ROI 配置
- 相机采图
- PLC 触发与 NG 输出
- 检测结果保存与追溯

因此，引入 AI 的合理方式不是推翻现有架构，而是在现有 ROI 检测链路中新增 AI 判定模块，形成“规则法 + AI”可并存的工业检测体系。

## 2. AI 训练的大致原理

AI 训练的本质，是让模型学习从输入图像到输出结论之间的映射关系。

在本项目中，输入通常是整张图像，或者更常见的是某个 ROI 裁剪区域；输出则是业务需要的检测结论，例如：

- 有垫片 / 无垫片
- 有涂胶 / 无涂胶
- 垫片或涂胶区域的位置与面积
- 缺陷概率或置信度

训练阶段，模型会反复读取大量“已知正确答案”的样本，通过不断调整内部参数，让预测结果逐渐接近人工标注结果。部署上线后，模型通常不再继续训练，而只负责推理，也就是对新图像输出结果。

### 2.1 常见 AI 任务形式

#### 1. 分类

输入一个 ROI 小图，输出一个类别结果。

适用场景：

- 只关心是否存在垫片
- 只关心是否存在涂胶
- 只需要输出 OK / NG 或有 / 无结论

#### 2. 分割

输入一个 ROI 小图，输出像素级区域掩膜。

适用场景：

- 既要判断有无，又要知道实际覆盖面积
- 需要判断涂胶连续性、断胶、溢胶范围
- 需要更细致的位置和面积量化

#### 3. 检测

输入整图，输出目标框位置。

适用场景：

- 目标位置变化较大
- 无法稳定使用固定 ROI
- 需要先找目标，再做后续判定

### 2.2 结合当前项目的推荐方向

结合当前业务和软件结构，第一阶段更建议采用 ROI 级分类，而不是直接做整图检测或像素级分割。

原因如下：

- 当前工位、视角和 ROI 相对固定
- 现有软件已经支持模板和 ROI 管理
- ROI 分类对数据量要求更低，训练速度更快
- 更容易和现有规则法进行并行验证
- 更适合工业项目的第一版快速落地

建议的第一版 AI 输出形式：

- 输入：ROI 图像
- 输出：二分类结果 + 置信度

例如：

- 有垫片，置信度 0.96
- 无垫片，置信度 0.08

## 3. 具体流程

引入 AI 训练建议按以下步骤推进。

### 3.1 第一步：明确训练目标

第一版必须先聚焦一个明确任务，不建议一开始同时做多种缺陷。

推荐优先选择以下其中一个：

- 判断 ROI 内是否存在垫片
- 判断 ROI 内是否存在涂胶

在任务定义阶段，需要先明确以下问题：

- 模型输入是整图还是 ROI
- 模型输出是二分类、三分类还是分割图
- 现场最终判定依据是什么
- 漏检和误检哪个风险更高

建议第一版采用：

- 输入：单个 ROI 图像
- 输出：有 / 无 的二分类结果
- 结果附带概率值

### 3.2 第二步：采集数据

数据质量通常比模型结构本身更重要。

需要尽量覆盖现场的真实波动，包括：

- 正常 OK 样本
- 真实 NG 样本
- 不同光照强度下的样本
- 不同批次、不同材质反光差异样本
- 轻微偏位样本
- 污染、脏污、反光异常样本
- 曝光和增益轻微波动样本

建议原则：

- 每一类至少准备 300 到 1000 个 ROI 样本
- NG 样本不足时，需要主动补采或模拟采集
- 训练数据必须覆盖真实现场波动，否则模型实验室表现好，现场效果会明显下降

当前软件已经具备结果图片保存能力，后续可以利用这套机制持续沉淀训练样本。

### 3.3 第三步：数据标注

如果第一版选择分类任务，标注方式相对简单，只需要给 ROI 图像打类别标签。

例如：

- gasket_present
- gasket_absent
- glue_present
- glue_absent

建议的数据目录结构如下：

```text
dataset/
  train/
    gasket_present/
    gasket_absent/
  val/
    gasket_present/
    gasket_absent/
  test/
    gasket_present/
    gasket_absent/
```

如果后续需要多产品、多工位扩展，建议从一开始保留以下元数据：

- 产品型号
- 工位号
- 相机编号
- 光源参数
- 班次或时间段
- 人工复判结果
- 最终确认标签

### 3.4 第四步：训练模型

第一版建议直接使用成熟的轻量分类网络，不建议自定义复杂网络结构。

推荐候选：

- ResNet18
- MobileNetV3
- EfficientNet-B0

对于当前项目，MobileNetV3 或 ResNet18 更适合作为第一版：

- 训练速度较快
- 推理速度较快
- Windows + Python 环境接入简单
- 更适合 ROI 小图二分类场景

训练时通常会做的基础处理包括：

- 统一输入尺寸，例如 224 x 224
- 归一化
- 数据增强

工业场景中的数据增强建议控制在合理范围内，例如：

- 轻微亮度变化
- 轻微对比度变化
- 轻微平移
- 轻微模糊
- 少量噪声扰动

不建议一开始使用与现场不符的增强方式，例如大角度旋转、上下翻转、左右翻转，除非现场真实存在这些变化。

### 3.5 第五步：验证模型

验证不能只看训练集准确率，需要同时关注验证集和独立测试集表现。

常用指标包括：

- Accuracy
- Precision
- Recall
- F1
- 混淆矩阵
- NG 漏检率
- OK 误杀率

工业项目里更重要的是业务风险，而不是单一总体准确率。

例如：

- 如果“无垫片”被误判成“有垫片”，这类漏检可能风险极高
- 如果 OK 被误报为 NG，虽然影响效率，但通常风险低于漏检

因此模型阈值不应只追求最高准确率，而要结合现场风险定义最终阈值。

### 3.6 第六步：导出模型

训练完成后，不建议直接把训练框架打包进产线软件。更稳妥的方式是将模型导出为标准推理格式。

优先推荐：

- ONNX

原因：

- 适合 Python 工业软件集成
- Windows 环境部署便利
- 可使用 ONNX Runtime 做稳定推理
- 更方便后续版本管理与替换

部署后的典型推理流程如下：

1. 相机采图
2. 按配方裁剪 ROI
3. 做必要预处理
4. 输入 ONNX 模型
5. 获取分类概率
6. 按阈值判定 OK / NG
7. 保存结果并输出 PLC

### 3.7 第七步：集成到现有软件

当前软件中每个 ROI 已经支持算法选择，因此后续最合理的接入方式，是在现有检测器中新增 AI 类型算法。

例如，将当前算法集合从：

- binary_gray_ratio
- ssim

扩展为：

- binary_gray_ratio
- ssim
- ai_classifier

如果后续做分割，还可以继续扩展：

- ai_segment

这样可以实现：

- 有些 ROI 继续使用规则法
- 有些 ROI 使用 AI 分类
- 不同配方按实际业务选择不同策略

这种“规则法 + AI”的混合策略，更适合工业软件分阶段落地。

## 4. 结合当前项目的推荐路线

### 4.1 第一版最小闭环建议

不建议一开始就做完整 AI 平台，建议先完成一个最小可验证闭环。

推荐第一版目标：

- 只针对一个 ROI
- 只做一个任务
- 只做二分类
- 只接一个模型

最适合的第一版示例：

- ROI 内是否有垫片

原因：

- 检测目标最明确
- 标注难度最低
- 输出最简单
- 最容易与当前 binary_gray_ratio 做对比

推荐的第一版闭环：

1. 使用当前软件保存 ROI 样本
2. 人工对 ROI 样本标注“有垫片 / 无垫片”
3. 训练 ROI 分类模型
4. 导出 ONNX 模型
5. 在现有检测器中新增 ai_classifier
6. 在配方中为 ROI 配置模型路径和判定阈值
7. 现场并行试运行，对比规则法和 AI 结果

### 4.2 为什么不建议第一阶段就做在线训练

第一阶段不建议在当前桌面软件里直接加入“点击按钮开始训练”的功能。

更合理的方式是：

- 训练在开发机或带显卡的训练机上完成
- 产线软件只负责推理

原因如下：

- 训练环境依赖重，不适合部署到现场机
- 训练过程不稳定，不适合和产线运行耦合
- 产线软件更强调稳定性、可追溯和快速恢复

因此第一阶段建议将“训练”和“运行”彻底分离。

## 5. 后续需要新增的模块

如果确定进入 AI 接入阶段，建议逐步新增以下模块。

### 5.1 数据集管理模块

职责：

- 从现有结果图或 ROI 图中整理训练数据
- 维护训练集、验证集、测试集划分
- 为样本保留元数据和追溯信息

### 5.2 标注规范与标签管理

职责：

- 明确标签定义
- 统一 OK / NG 和各类缺陷命名
- 规范人工复判与修正流程

### 5.3 训练脚本或训练工程

职责：

- 数据加载
- 模型训练
- 验证评估
- 模型导出

建议与产线软件分目录维护，必要时甚至独立仓库维护。

### 5.4 推理模块

职责：

- 加载 ONNX 模型
- 执行预处理和推理
- 输出分类结果和置信度
- 对接现有 ROI 判定流程

### 5.5 配方扩展

每个 ROI 后续建议支持以下 AI 配置项：

- 算法类型
- 模型文件路径
- 类别定义
- 判定阈值
- 输入尺寸
- 图像预处理参数
- 模型版本号

### 5.6 结果追溯扩展

建议在结果保存中增加：

- 模型名称
- 模型版本
- 推理概率
- 当前阈值
- 最终判定结果
- 规则法与 AI 并行结果对比

## 6. 分阶段实施规划

### 阶段 1：任务定义

目标：

- 明确第一版 AI 只解决一个单一问题

建议输出：

- 第一版任务定义说明
- 输入输出定义
- 风险优先级说明

推荐任务：

- 单 ROI 垫片有无二分类

### 阶段 2：数据准备

目标：

- 基于现有保存图建立第一版训练样本库

建议输出：

- 数据目录结构
- 样本筛选规则
- 样本数量统计
- 数据集划分方案

### 阶段 3：标注与复核

目标：

- 建立统一标签体系，完成第一版样本标注

建议输出：

- 标签规范
- 标注记录
- 人工复核结果

### 阶段 4：训练与验证

目标：

- 训练第一版分类模型并输出评估指标

建议输出：

- 训练脚本
- 模型文件
- 验证报告
- 混淆矩阵
- 阈值建议

### 阶段 5：部署接入

目标：

- 将模型导出为 ONNX 并接入当前检测软件

建议输出：

- 推理模块
- 配方扩展字段
- ROI 算法新增 ai_classifier
- 部署说明

### 阶段 6：现场并行试运行

目标：

- 不立即替换规则法，而是先并行运行并比较差异

建议输出：

- AI 与规则法对比报表
- 差异样本清单
- 复判结论
- 阈值修订建议

### 阶段 7：正式切换与版本管理

目标：

- 完成现场验证后，决定是否将 AI 作为主判定手段

建议输出：

- 模型版本管理规则
- 回滚机制
- 上线审批记录
- 切换标准与验收标准

## 7. 推荐的工程策略

### 7.1 采用规则法与 AI 并行的混合策略

建议不要立即废弃现有规则法算法，而应采用以下策略：

- 规则法继续保留，作为对照和兜底
- AI 先做并行判定，不直接控制最终上线输出
- 待 AI 稳定后，再评估是否切换主判定策略

这种方式更适合工业软件逐步上线，可以降低误切换风险。

### 7.2 先做单点突破，再扩展多产品

建议顺序：

1. 先做单产品、单工位、单 ROI
2. 再扩展到多 ROI
3. 再扩展到多配方、多产品、多工位

不要一开始就同时做：

- 多种缺陷
- 多模型并存
- 在线训练
- 自动阈值学习

这样会显著增加项目复杂度和失控风险。

## 8. 当前阶段的建议结论

结合当前项目现状，最推荐的下一步不是直接改现有产线代码，而是先完成 AI 接入方案定型。

建议优先做两件事：

1. 输出 AI 检测接入方案设计
2. 输出分阶段实施规划和里程碑

如果后续正式进入实现阶段，建议的优先顺序为：

1. 定义第一版任务
2. 整理和补采数据
3. 建立标注规范
4. 训练并验证第一版模型
5. 导出 ONNX 并接入现有检测器
6. 现场并行验证
7. 决定是否切换主判定

## 9. 第一版建议摘要

为了便于快速决策，这里给出第一版最推荐方案摘要：

- 第一版任务：单 ROI 垫片有无二分类
- 模型形式：ROI 分类模型
- 模型候选：MobileNetV3 或 ResNet18
- 部署格式：ONNX
- 集成方式：在现有 ROI 算法中新增 ai_classifier
- 上线方式：先并行，不直接替换规则法
- 训练部署策略：训练与产线运行分离

以上方案能够最大限度复用当前项目已有的配方、ROI、保存、PLC 与结果追溯架构，风险较低，便于快速形成第一版 AI 检测闭环。
