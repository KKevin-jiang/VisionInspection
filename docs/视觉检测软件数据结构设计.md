# 视觉检测软件数据结构设计

## 1. 文档目的

本文档用于定义视觉检测软件一期的核心数据结构，作为后续界面开发、业务逻辑开发、配置文件读写和结果存档的统一依据。

本文档覆盖以下对象：

1. 配方文件结构
2. 模板结构
3. ROI 结构
4. 本地检测记录文件结构
5. 推荐的本地文件目录结构

设计原则如下：

1. 优先满足一期单工位单相机、单配方单模板、多 ROI 的业务需求。
2. 数据结构必须兼容后续一个配方多个模板的扩展。
3. 数据结构必须兼容后续多工位相机扩展。
4. 配置文件与业务代码解耦，便于通过界面进行编辑和保存。
5. 记录文件应同时兼顾人可读性和程序可解析性。

## 2. 总体设计原则

### 2.1 文件格式选择

建议采用以下格式：

1. 配方文件使用 JSON。
2. 检测记录主文件使用 JSON Lines 或 JSON。
3. 统计汇总文件可选 CSV。
4. 模板图、原图、结果图使用图片文件保存。

原因如下：

1. JSON 适合层级结构，如配方、模板、ROI、PLC 参数。
2. JSON 易于 Python 直接读写。
3. JSON 适合后续扩展字段，不容易因版本升级破坏旧结构。
4. CSV 适合后续导出给生产、品质或主管查看。

### 2.2 标识规则

所有关键对象建议同时具备以下标识：

1. 内部唯一编号 id
2. 对用户可见的 name
3. 可选的 code 或编号字段

建议 id 使用字符串，便于后续切换为 UUID 或自定义编码。

### 2.3 时间字段规则

所有时间字段建议采用 ISO 8601 格式，例如：

2026-05-18T14:25:30.125+08:00

### 2.4 版本字段规则

所有核心配置文件建议包含：

1. schema_version：数据结构版本
2. app_version：生成该文件的软件版本

便于后续升级时做兼容处理。

## 3. 配方文件结构设计

### 3.1 配方对象职责

配方是业务核心对象，用于描述某个产品或某类产品的完整检测规则。

一个配方至少负责管理：

1. 基本信息
2. 工位与相机绑定关系
3. 模板列表
4. ROI 配置
5. 判定规则
6. 图像保存规则
7. PLC 联动参数
8. 运行参数

### 3.2 配方顶层结构

建议一个配方文件对应一个 JSON 文件，顶层结构如下：

```json
{
  "schema_version": "1.0",
  "app_version": "1.0.0",
  "recipe": {}
}
```

### 3.3 配方字段定义

recipe 对象建议包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 配方唯一标识 |
| code | string | 否 | 配方编号，可用于现场编码 |
| name | string | 是 | 配方名称 |
| product_name | string | 是 | 产品名称 |
| product_model | string | 否 | 产品型号 |
| description | string | 否 | 配方说明 |
| station_id | string | 是 | 工位编号 |
| camera_id | string | 是 | 绑定相机编号 |
| enabled | bool | 是 | 配方是否启用 |
| trigger_mode | string | 是 | 触发模式，当前固定为 plc_external |
| template_match_mode | string | 是 | 模板匹配模式，当前建议 single_active 或 best_match |
| templates | array | 是 | 模板列表 |
| decision_policy | object | 是 | 判定策略 |
| preprocess | object | 是 | 预处理参数 |
| plc | object | 是 | PLC 联动参数 |
| storage | object | 是 | 本地保存参数 |
| runtime | object | 否 | 运行期参数 |
| created_at | string | 是 | 创建时间 |
| updated_at | string | 是 | 更新时间 |
| created_by | string | 否 | 创建人 |
| updated_by | string | 否 | 修改人 |
| tags | array | 否 | 标签列表 |

### 3.4 判定策略结构

decision_policy 建议结构如下：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| mode | string | 是 | 判定模式，当前建议 all_roi_pass |
| min_pass_count | int | 否 | 最小通过 ROI 数，当前模式下可为空 |
| allow_disabled_roi | bool | 是 | 是否允许 ROI 禁用 |
| final_ng_on_any_fail | bool | 是 | 任一 ROI 失败即 NG |

一期建议固定为：

1. mode = all_roi_pass
2. final_ng_on_any_fail = true

### 3.5 预处理参数结构

preprocess 建议结构如下：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| grayscale | bool | 是 | 是否灰度化 |
| resize_mode | string | 是 | 尺寸处理模式，建议 keep_roi_size |
| denoise_enabled | bool | 是 | 是否降噪 |
| denoise_method | string | 否 | 降噪方法 |
| normalize_enabled | bool | 是 | 是否做亮度归一化 |
| blur_kernel | int | 否 | 模糊核大小 |

### 3.6 PLC 参数结构

plc 建议结构如下：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| enabled | bool | 是 | 是否启用 PLC 联动 |
| trigger_source | string | 是 | 触发来源，当前固定 plc |
| protocol | string | 否 | 协议类型，后续根据现场确定 |
| connection | object | 否 | 连接参数 |
| ng_output | object | 是 | NG 输出参数 |
| heartbeat_enabled | bool | 否 | 是否启用心跳 |
| timeout_ms | int | 否 | 通讯超时 |

ng_output 建议包含：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| enabled | bool | 是 | 是否启用 NG 输出 |
| signal_name | string | 否 | 信号名称 |
| channel | string | 否 | 输出通道 |
| pulse_ms | int | 是 | 脉冲保持时间 |
| delay_ms | int | 是 | 输出延时 |
| reset_mode | string | 是 | 复位方式 |

### 3.7 本地保存参数结构

storage 建议结构如下：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| root_dir | string | 是 | 数据根目录 |

| save_result_image | bool | 是 | 是否保存结果图 |
| save_only_ng_image | bool | 是 | 是否仅保存 NG 图 |

| date_subdir_mode | string | 是 | 是否按日期分目录 |
| max_retention_days | int | 否 | 最大保留天数 |

### 3.8 运行参数结构

runtime 建议结构如下：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| target_cycle_ms | int | 是 | 目标节拍，当前应小于 1000 |
| detection_timeout_ms | int | 是 | 单次检测超时 |
| retry_on_capture_fail | int | 否 | 采图失败重试次数 |
| allow_manual_test | bool | 是 | 是否允许手动测试 |

## 4. 模板结构设计

### 4.1 模板对象职责

模板对象用于描述一个可参与检测的标准图像及其 ROI 集合。

一个配方下可以包含多个模板，用于后续扩展：

1. 同一产品不同外观版本
2. 同一产品不同姿态
3. 同一产品不同批次参考图

### 4.2 模板字段定义

templates 数组中的每个模板对象建议包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 模板唯一标识 |
| code | string | 否 | 模板编号 |
| name | string | 是 | 模板名称 |
| enabled | bool | 是 | 是否启用 |
| is_default | bool | 是 | 是否默认模板 |
| image_path | string | 是 | 模板图像路径 |
| image_width | int | 是 | 模板图宽度 |
| image_height | int | 是 | 模板图高度 |
| image_checksum | string | 否 | 模板图校验值 |
| match_priority | int | 否 | 匹配优先级 |
| roi_list | array | 是 | ROI 列表 |
| description | string | 否 | 说明 |
| tags | array | 否 | 标签 |
| created_at | string | 是 | 创建时间 |
| updated_at | string | 是 | 更新时间 |

### 4.3 模板设计说明

1. image_path 应优先使用相对路径，便于整体搬迁项目目录。
2. image_width 与 image_height 用于校验模板与 ROI 坐标合法性。
3. image_checksum 可用于判断模板文件是否被替换。
4. roi_list 直接挂在模板下，便于一个模板拥有独立 ROI 集合。

### 4.4 模板 JSON 示例

```json
    "default"

  "updated_at": "2026-05-18T10:00:00+08:00"
```

## 5. ROI 结构设计

### 5.1 ROI 对象职责

ROI 是最核心的检测单元，用于描述模板图中需要参与比对的区域及其阈值规则。

### 5.2 ROI 字段定义

roi_list 数组中的每个 ROI 对象建议包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | ROI 唯一标识 |
| index | int | 是 | ROI 显示顺序 |
| name | string | 是 | ROI 名称 |
| enabled | bool | 是 | 是否启用 |
| shape | string | 是 | 形状，当前固定 rectangle |
| x | int | 是 | 左上角 x 坐标 |
| y | int | 是 | 左上角 y 坐标 |
| width | int | 是 | 宽度 |
| height | int | 是 | 高度 |
| threshold | float | 是 | 相似度阈值 |
| algorithm | string | 是 | 相似度算法 |
| score_weight | float | 否 | 权重，后续扩展用 |
| fail_color | string | 否 | NG 时显示颜色 |
| pass_color | string | 否 | OK 时显示颜色 |
| description | string | 否 | 说明 |
| created_at | string | 是 | 创建时间 |
| updated_at | string | 是 | 更新时间 |

### 5.3 ROI 设计约束

1. 一期 shape 固定为 rectangle。
2. x、y、width、height 必须在模板图尺寸范围内。
3. threshold 建议取值范围为 0 到 1。
4. algorithm 一期建议固定为 ssim，后续可扩展为 template_match、histogram 等。

### 5.4 ROI JSON 示例

```json
  "width": 260,
  "height": 180,
  "threshold": 0.92,
  "algorithm": "ssim",
  "score_weight": 1.0,
  "fail_color": "#ff3b30",
  "pass_color": "#22c55e",
  "description": "品牌 Logo 检测区域",
  "created_at": "2026-05-18T10:00:00+08:00",

}
```

## 6. 本地检测记录文件结构设计

### 6.1 记录对象职责

检测记录用于保存每次触发后的完整结果，用于追溯、排查、抽检和数据统计。

检测记录应包括：

1. 触发上下文
2. 配方和模板信息
3. 图像保存路径
4. 每个 ROI 的检测结果
5. 最终判定
6. PLC 输出情况
7. 异常信息
8. 性能耗时信息

| template_name | string | 是 | 模板名称 |
| trigger_source | string | 是 | 触发来源 |

| result_image_path | string | 否 | 结果图路径 |

## 7. 本地汇总记录文件设计

### 7.1 汇总记录用途

除了单次 JSON 记录，建议另外维护一个 CSV 汇总文件，便于用户直接查看和筛选。

### 7.2 CSV 建议字段

建议 CSV 至少包含以下列：

1. record_id
2. triggered_at
3. station_id
4. recipe_name
5. template_name
6. overall_result
7. cycle_time_ms
8. ng_roi_names
9. raw_image_path
10. result_image_path

### 7.3 设计说明

1. CSV 只保存摘要信息。
2. 详细信息仍以单次 JSON 记录为准。
3. 若后续需要数据追溯或分析，可直接读取 JSON 详情。

## 8. 推荐的本地目录结构

建议项目数据目录使用以下结构：

```text
data/
  recipes/
    recipe-cap-001.json
  templates/
    recipe-cap-001/
      template_a.png
      template_b.png
  records/
    2026-05-18/
      ST01/
        raw/
          20260518-ST01-000001.png
        result/
          20260518-ST01-000001_result.png
        json/
          20260518-ST01-000001.json
        summary.csv
  logs/
    app.log
```

### 8.1 目录说明

1. recipes 目录保存配方 JSON。
2. templates 目录按配方分模板图文件夹。
3. records 目录按日期和工位分层。
4. raw 保存原图。
5. result 保存带标注结果图。
6. json 保存单次检测详细记录。
7. summary.csv 保存每日或每工位摘要。
8. logs 保存程序运行日志。

## 9. 一期开发落地建议

### 9.1 一期必须实现的数据结构

一期至少应实现：

1. 一个配方 JSON 文件。
2. 一个模板对象。
3. 多个 ROI 对象。
4. 每次检测一个 JSON 记录文件。
5. 一个 CSV 汇总文件。

### 9.2 一期可先简化但不要删掉的字段

以下字段即使一期暂时未完全使用，也建议保留：

1. templates 数组
2. match_priority
3. score_weight
4. plc.protocol
5. runtime
6. schema_version

原因是这些字段会直接影响后续多模板、多工位和版本兼容。

## 10. 字段命名建议

建议统一采用以下命名规则：

1. JSON 字段使用 snake_case。
2. 布尔值字段尽量以 is、has、enabled、allow 开头。
3. 时间字段统一使用 _at 结尾。
4. 耗时字段统一使用 _ms 结尾。
5. 路径字段统一使用 _path 结尾。

## 11. 后续扩展建议

后续如扩展多模板和多工位，可在当前结构基础上直接增加：

1. 一个工位绑定多个相机的 station 配置层。
2. 模板筛选条件字段。
3. ROI 分组字段。
4. 配方切换条件字段。
5. 统计结果文件或数据库桥接层。

## 12. 结论

以上数据结构设计能够满足当前项目的一期需求，并且保留了多模板、多工位、算法扩展和记录扩展能力。后续进入代码实现时，建议先按本文档定义数据类、JSON 读写器和目录管理逻辑，再开始 UI 与检测流程开发。
