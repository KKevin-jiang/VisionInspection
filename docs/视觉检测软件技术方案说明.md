# 视觉检测软件技术方案说明

## 1. 文档目的

本文档用于将现有需求、数据结构设计和界面规划转化为可直接进入开发阶段的技术方案，作为项目实施、模块拆分、开发排期和代码落地的依据。

本文档重点说明以下内容：

1. 总体技术架构
2. 项目目录规划
3. 模块拆分方案
4. 线程模型
5. 相机采图实现方案
6. OpenCV 检测实现方案
7. PLC 联动实现方案
8. 数据流与时序
9. 异常处理和日志策略
10. 一期实施建议

## 2. 建设目标

系统应基于海康机器人工业相机 Python SDK 完成以下闭环：

1. 加载产品检测配方。
2. 连接相机并进入 PLC 外部触发待机状态。
3. 接收到 PLC 触发信号后采集一帧图像。
4. 使用 OpenCV 对图像执行预处理与 ROI 相似度检测。
5. 输出每个 ROI 的分值和最终 OK 或 NG。
6. NG 时在界面高亮区域并通过 PLC 输出 NG 信号。
7. 保存图像和检测记录到本地文件。
8. 保证单次检测总节拍小于 1 秒。

## 3. 技术选型建议

### 3.1 开发语言与框架

建议技术栈如下：

1. Python 3.10 或 3.11
2. PyQt5 用于桌面界面开发
3. OpenCV 用于图像处理
4. NumPy 用于图像数组和数值处理
5. scikit-image 用于 SSIM 计算
6. 标准库 json、csv、pathlib、threading、queue、logging 用于配置、记录和线程协作

### 3.2 相机 SDK

相机层基于当前工作区已有海康 SDK 示例改造，核心参考来源包括：

1. [AreaScanCamera/BasicDemo/BasicDemo.py](AreaScanCamera/BasicDemo/BasicDemo.py)
2. [AreaScanCamera/BasicDemo/CamOperation_class.py](AreaScanCamera/BasicDemo/CamOperation_class.py)

建议保留原 SDK 接入方式，但不要继续把业务逻辑堆在示例脚本内，而是抽取为独立模块。

### 3.3 PLC 通讯

由于现场已确认由 PLC 提供触发并接收 NG 输出，但协议未最终确定，因此建议采用适配层设计。

一期技术方案建议：

1. 抽象统一 PLC 接口。
2. 先实现一个模拟适配器用于联调。
3. 现场协议确定后再补真实适配器。

建议接口支持以下实现方式之一：

1. TCP Socket
2. Modbus TCP
3. 串口通讯
4. 数字 IO 中间层

## 6. 模块拆分方案

### 6.1 主控流程模块

核心模块建议为 inspection_controller 和 inspection_service。

#### inspection_controller 职责

1. 监听 PLC 触发事件。
2. 调度采图、检测、结果展示和记录保存。
3. 管理系统状态流转。
4. 将结果回传给 UI。

#### inspection_service 职责

1. 调用相机采图。
2. 调用检测模块执行比对。
3. 调用判定逻辑生成最终结果。
4. 调用记录服务和 PLC 输出服务。

### 6.2 配方管理模块

建议拆成两部分：

1. recipe_repository
2. recipe_service

#### recipe_repository

负责：

1. 从 JSON 加载配方
2. 保存配方 JSON
3. 校验文件路径和目录

#### recipe_service

负责：

1. 当前配方切换
2. 当前模板切换
3. ROI 合法性校验
4. 配方编辑过程的数据整合

### 6.3 相机模块

建议分成三个核心文件：

1. hik_camera_client.py
2. frame_converter.py
3. camera_worker.py

#### hik_camera_client

负责：

1. 调用海康 SDK 枚举设备
2. 创建句柄
3. 打开和关闭设备
4. 设置外部触发模式
5. 获取图像缓冲
6. 释放图像缓冲

#### frame_converter

负责：

1. 将 SDK 原始缓冲区转为 NumPy 数组
2. 将不同像素格式统一转为 OpenCV 可处理格式
3. 输出灰度图或 BGR 图像

#### camera_worker

负责：

1. 在独立线程内等待采图请求
2. 执行单帧采集
3. 将图像结果投递到检测线程

### 6.4 视觉检测模块

建议拆分为：

1. preprocess.py
2. similarity.py
3. detector.py

#### preprocess.py

负责：

1. 灰度化
2. 尺寸裁剪
3. 归一化
4. 去噪
5. 输出适合比对的 ROI 图像

#### similarity.py

负责：

1. SSIM 计算
2. 模板匹配分值计算
3. 后续扩展直方图或其他算法

#### detector.py

负责：

1. 接收整张图和模板对象
2. 按 ROI 截取待检区域
3. 调用 preprocess 和 similarity
4. 汇总 ROI 结果
5. 输出 InspectionResult 对象

### 6.5 PLC 模块

建议采用统一抽象基类。

#### plc_base.py

定义统一接口：

1. connect
2. disconnect
3. is_connected
4. wait_for_trigger
5. send_ng_signal
6. reset_output

#### plc_mock.py

用于：

1. 本地联调
2. 无 PLC 时验证完整流程
3. 自动化测试模拟触发

#### plc_tcp_adapter.py

后续在明确协议后用于对接真实 PLC。

### 6.6 记录保存模块

建议拆为：

1. record_service
2. file_manager
3. image_saver
4. json_writer
5. csv_writer

职责如下：

1. 负责生成记录编号
2. 负责目录创建
3. 负责原图和结果图保存
4. 负责 JSON 明细记录写入
5. 负责 CSV 摘要写入

## 7. 线程模型设计

### 7.1 线程模型目标

线程设计需要满足以下目标：

1. 保证 UI 不被检测和保存操作阻塞。
2. 保证 PLC 触发响应及时。
3. 保证相机采图和图像检测流程稳定。
4. 满足单次检测节拍小于 1 秒。

### 7.2 建议线程划分

建议最少使用四类执行单元：

1. UI 主线程
2. PLC 监听线程
3. 检测工作线程
4. 结果保存线程

可选增加：

1. 相机独立工作线程

### 7.3 各线程职责

#### UI 主线程

负责：

1. 界面渲染
2. 用户交互
3. 状态显示
4. 图像展示

UI 主线程不直接执行以下操作：

1. 相机阻塞采图
2. PLC 阻塞监听
3. OpenCV 批量检测
4. 文件保存

#### PLC 监听线程

负责：

1. 持续监听 PLC 触发信号
2. 触发后向检测流程投递任务

建议特点：

1. 轻量阻塞式或轮询式实现
2. 不做复杂业务处理

#### 检测工作线程

负责：

1. 接收一次检测任务
2. 调用相机采图
3. 执行预处理和 ROI 检测
4. 生成结果对象
5. 通知 UI 和保存线程
6. 执行 PLC NG 输出

#### 结果保存线程

负责：

1. 保存原图
2. 保存结果图
3. 写 JSON 和 CSV
4. 清理旧文件

### 7.4 线程间通信建议

建议使用以下机制：

1. queue.Queue 用于任务和结果传递
2. PyQt Signal 用于通知 UI 更新
3. threading.Event 用于状态同步和停止控制

### 7.5 建议的任务流

```text
PLC监听线程 -> 触发检测任务队列 -> 检测工作线程
检测工作线程 -> UI信号 -> 主界面更新
检测工作线程 -> 保存任务队列 -> 结果保存线程
检测工作线程 -> PLC输出接口 -> 发送NG信号
```

### 7.6 是否并发处理多次触发

一期建议采取串行检测策略：

1. 同一工位只允许一个检测任务在执行中。
2. 若上一任务未结束又收到触发，可配置为丢弃、排队或报警。

建议一期默认策略：

1. 系统忙时忽略新触发并记录告警。

原因：

1. 逻辑更清晰
2. 更符合单工位现场节拍
3. 避免相机和 PLC 状态竞争

## 8. 相机采图实现方案

### 8.1 实现目标

基于海康 SDK 封装一个稳定的单帧采图能力，供检测流程直接调用。

### 8.2 改造思路

当前示例代码以演示连续取流和 GUI 展示为主，不适合直接作为业务核心。建议改造原则如下：

1. 抽离设备打开和关闭逻辑。
2. 抽离触发模式设置逻辑。
3. 抽离单次取图接口。
4. 抽离图像缓冲转换逻辑。

### 8.3 相机客户端推荐接口

建议相机客户端提供如下方法：

1. initialize_sdk
2. finalize_sdk
3. enum_devices
4. open_device
5. close_device
6. set_external_trigger_mode
7. start_grabbing
8. stop_grabbing
9. grab_one_frame

### 8.4 推荐采图模式

由于现场由 PLC 外部触发，建议模式如下：

1. 相机进入外部触发模式。
2. 保持取流开启状态。
3. 触发到来后调用获取一帧图像接口。

建议不要每次触发都重新打开和关闭相机，因为这会影响稳定性和节拍。

### 8.5 图像格式转换

相机 SDK 返回的是底层缓冲区，需要转换为 OpenCV 可处理的 NumPy 图像。

转换步骤建议如下：

1. 获取帧宽、高、像素类型。
2. 将原始缓冲区拷贝到 Python 可管理内存。
3. 根据像素格式转为灰度或 BGR。
4. 返回 ndarray。

### 8.6 采图异常处理

至少处理以下异常：

1. 相机未打开
2. 获取缓冲区超时
3. 图像格式不支持
4. 断线
5. 缓冲释放失败

## 9. OpenCV 检测实现方案

### 9.1 检测目标

基于模板图和多 ROI 结构，输出每个 ROI 的相似度和最终 OK 或 NG。

### 9.2 一期检测流程

建议流程如下：

1. 获取当前配方和当前模板。
2. 对采集图像执行预处理。
3. 遍历模板下的启用 ROI。
4. 在采集图像中按 ROI 坐标截取待检区域。
5. 在模板图中截取对应模板区域。
6. 对两个区域执行相似度计算。
7. 输出单 ROI 分值。
8. 根据阈值判定单 ROI 是否通过。
9. 汇总全部 ROI 结果得到最终结论。

### 9.3 预处理建议

一期预处理建议保持简单稳定：

1. 转灰度
2. 必要时高斯滤波
3. 可选直方图归一化
4. 保持 ROI 原尺寸，不额外缩放整图

### 9.4 相似度算法建议

一期主算法建议优先采用 SSIM。

原因如下：

1. 对轻微亮度变化更稳定。
2. 业务语义更接近“相似度”。
3. 适合固定 ROI 场景。

同时建议保留 template_match 作为备用算法接口。

### 9.5 SSIM 实现建议

建议流程：

1. 将模板 ROI 与待检 ROI 转为灰度图。
2. 保证两者尺寸一致。
3. 调用 skimage.metrics.structural_similarity。
4. 将输出分值标准化为 0 到 1。

### 9.6 结果对象设计

检测流程建议输出统一 InspectionResult 对象，包含：

1. 原图
2. 结果图
3. ROI 结果列表
4. NG ROI 列表
5. 总结果
6. 耗时信息
7. 异常信息

### 9.7 结果标注实现

结果图生成建议使用 OpenCV 在图像上绘制：

1. ROI 边框
2. ROI 名称
3. 分值文本
4. OK 或 NG 颜色区分

NG ROI 建议用红框，OK ROI 建议用绿框。

## 10. PLC 联动实现方案

### 10.1 目标

实现两类 PLC 联动能力：

1. 接收 PLC 外部触发
2. 输出 PLC NG 信号

### 10.2 抽象接口设计

建议定义 PlcBase 接口：

1. connect
2. disconnect
3. wait_for_trigger
4. send_ng_signal
5. clear_ng_signal
6. health_check

### 10.3 触发实现策略

依据现场协议不同，可以有两种主流实现：

1. 轮询读取 PLC 指定位
2. 阻塞接收 PLC 消息

一期建议架构上兼容两种模式，但代码接口对上层保持一致。

### 10.4 NG 输出实现策略

建议流程如下：

1. 检测结果为 NG。
2. 调用 send_ng_signal。
3. 按配置等待 pulse_ms。
4. 自动复位或调用 clear_ng_signal。
5. 返回输出成功或失败状态。

### 10.5 PLC 异常处理

至少处理以下问题：

1. 未连接
2. 连接超时
3. 触发读取失败
4. 输出失败
5. 复位失败

### 10.6 本地调试方案

在真实 PLC 未接入前，建议使用 plc_mock：

1. 按键触发模拟
2. 定时自动触发模拟
3. 记录 NG 输出调用日志

这样可以先把 UI、相机、检测和记录流程全部打通。

## 11. 系统状态机建议

建议定义统一系统状态，避免界面、线程和业务各自维护一套状态。

建议状态如下：

1. INIT
2. IDLE
3. WAIT_TRIGGER
4. CAPTURING
5. DETECTING
6. SAVING
7. NG_OUTPUTTING
8. ERROR

### 11.1 状态流转建议

```text
INIT -> IDLE -> WAIT_TRIGGER -> CAPTURING -> DETECTING -> SAVING -> WAIT_TRIGGER
                                        -> NG_OUTPUTTING -> SAVING -> WAIT_TRIGGER
任意状态异常 -> ERROR
ERROR恢复后 -> IDLE 或 WAIT_TRIGGER
```

## 12. 数据流与时序

### 12.1 单次检测数据流

```text
PLC触发
  -> 检测控制器
  -> 相机模块采图
  -> 图像转 ndarray
  -> OpenCV 检测模块
  -> 生成 InspectionResult
  -> UI 更新
  -> NG 时 PLC 输出
  -> 保存图片和记录
  -> 返回等待触发状态
```

### 12.2 单次检测时序建议

建议目标耗时分配如下：

1. PLC 触发识别：小于 20 ms
2. 相机取图：小于 150 ms
3. 图像转换：小于 50 ms
4. 检测计算：小于 300 ms
5. NG 输出：小于 50 ms
6. 记录保存：异步执行，不阻塞主流程

## 13. 异常处理与日志策略

### 13.1 日志级别建议

建议至少使用四级日志：

1. INFO
2. WARNING
3. ERROR
4. DEBUG

### 13.2 关键日志点

必须记录以下事件：

1. 软件启动和关闭
2. 相机连接和断开
3. PLC 连接和断开
4. 触发到达
5. 单次检测开始和结束
6. ROI 检测结果
7. 图片保存结果
8. PLC NG 输出结果
9. 异常堆栈

### 13.3 错误处理原则

1. 硬件异常与业务异常分开记录。
2. 检测失败时不要导致整个程序崩溃。
3. UI 应看到明确错误提示，而不是只在控制台打印。
4. 关键异常要写入日志文件和结果记录。

最小可用版本建议先完成：

## 16. 风险与缓解建议

### 16.1 相机 SDK 风险

风险：示例代码线程处理较粗糙，直接沿用可能引发稳定性问题。

缓解：独立封装相机客户端，不复用示例里的强制停线程做法作为业务主逻辑。

### 16.2 PLC 协议不确定风险

风险：真实接口未定，可能影响联调周期。

缓解：先做 PLC 抽象接口和 mock 适配器。

### 16.3 图像判定稳定性风险

风险：实际现场可能存在轻微位置偏移或亮度波动。

缓解：一期先记录分值和样本，必要时二期增加定位补偿。

### 16.4 节拍风险

风险：同步保存图片或 PLC 阻塞可能影响总节拍。

缓解：记录保存异步化，PLC 输出接口做超时控制。

## 17. 结论

本技术方案建议采用模块化、分层化和适配器式架构，以现有海康 SDK 示例为设备接入基础，通过独立的相机封装、OpenCV 检测模块、PLC 适配层和异步记录保存机制，完成一期视觉检测软件开发。该方案能够满足当前单工位、多 ROI、PLC 触发、NG 输出和本地留档需求，并为后续多模板和多工位扩展保留清晰的技术路径。
