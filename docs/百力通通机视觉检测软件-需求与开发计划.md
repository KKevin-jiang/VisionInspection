## 四、硬件 IO 说明与联调

### 4.1 接线确认

```
三菱 PLC ──── Line0（输入）──→ 工业相机（触发采集）
工业相机 ──── Line1（输出）──→ 三菱 PLC（合格/不合格信号）

Line1 逻辑：
  合格   → 高电平，持续 500ms（默认，可配置）后恢复低电平
  不合格 → 保持低电平
```

| 步骤 | 检查项 | 说明 |
|------|--------|------|
| 1 | PLC 触发线 → 相机 `Line0+` | 确认接在相机对应输入端，公共地和电平规格与相机型号一致 |
| 2 | 相机 `Line1+` → PLC 输入端 | NG/OK 回传线，确认 PLC 输入端与相机输出类型匹配 |
| 3 | IO 电平与供电 | 核对相机说明书：IO 电平标准、是否需要外接电源、是否为光耦输入输出 |

### 4.2 相机节点配置（MVS 客户端）

| 步骤 | 节点 | 设定值 | 说明 |
|------|------|--------|------|
| 1 | `TriggerMode` | `On` | 启用外部触发 |
| 2 | `TriggerSource` | `Line0` | 触发源为 Line0 输入 |
| 3 | `TriggerActivation` | `RisingEdge` | 与 PLC 输出边沿一致，通常为上升沿 |
| 4 | `LineDebouncerTime` | `50 µs` | 防抖滤波，防止误触发 |
| 5 | `LineSelector` | `Line1` → `LineMode = Output` | Line1 设为输出模式 |
| 6 | `LineSource` | `UserOutput0` | 映射到用户可编程输出节点（视相机型号而定） |

### 4.3 软件侧 Line0/Line1 参数

**Line0 触发参数**：
- `TriggerMode = On`
- `TriggerSource = Line0`
- `TriggerActivation = RisingEdge`
- `LineDebouncerTime = 50 µs`

**Line1 输出控制**（MVS SDK）：
- `LineSelector = Line1` → `LineMode = Output`
- 通过 `UserOutputSelector = UserOutput0` + `UserOutputValue = True/False` 控制电平

**配方侧检查**：
- 配方 `trigger_mode` 设为 `plc_external`
- 在配方编辑页"检测与IO"中确认 `相机输出线` 为 `Line1`，脉宽与延时符合现场节拍
- 主界面启动 IO 监听后，状态应停留在"等待 Line0"
- PLC 发出一个有效脉冲后，软件应立即进入检测并刷新图像与结果

### 4.4 结果回传验证

| 步骤 | 测试内容 | 验收标准 |
|------|----------|----------|
| 1 | 制造 OK 样本 | 确认检测完成后 Line1 **有脉冲输出**（高电平 ~500ms 后恢复低电平） |
| 2 | 制造 NG 样本 | 确认检测完成后 Line1 **保持低电平，无脉冲输出** |
| 3 | 脉宽测量 | 用示波器或 PLC 监控，脉宽与设定值一致（默认 500ms） |
| 4 | 无信号排查 | 若 PLC 收不到信号，优先检查 `LineSource`、输出类型、电平和公共地 |

### 4.5 常见 IO 故障排查

| 故障现象 | 可能原因 | 排查步骤 |
|----------|----------|----------|
| 一直等待触发 | `TriggerSource` 未设为 `Line0` | 1. 检查 MVS 中 TriggerSource 配置<br>2. 检查 PLC 脉冲宽度是否过窄 |
| 相机能采图但软件不更新 | 未启动 IO 监听或相机被其他程序占用 | 1. 确认主界面已启动 IO 监听<br>2. 关闭 MVS 客户端等其他占用相机的程序 |
| NG 不回传 / 无 Line1 输出 | Line1 输出未正确配置 | 1. 查 `Line1` 是否支持输出模式<br>2. 查 `UserOutput0` 是否被该相机型号支持<br>3. 查 `LineSource` 映射是否正确 |
| 节拍波动大 | 多种因素叠加 | 1. 排除 PLC 触发抖动<br>2. 检查曝光时间是否过长<br>3. 检查网络丢包<br>4. 检查磁盘保存是否阻塞 |

### 4.6 Line1 信号输出实际机制与调试记录（2026-06-12）

> **结论先行**：本相机 Line1 **不支持** `UserOutput`（软件直接控制电平）和 `TimerActive`（定时器脉冲），只能通过 **ExposureStartActive + StrobeEnable**（曝光同步频闪）输出脉冲。这意味着要产生 Line1 脉冲必须让相机真正曝光一次。

#### 4.6.1 相机 Line1 能力诊断

使用 `diagnose_output_line("Line1")` 获取的相机实际能力：

```
LineMode 可选=Strobe                    ← 只有频闪模式，不能设为普通 Output
LineSource 可选=ExposureStartActive,
    AcquisitionStartActive, ... 等     ← 无 UserOutput0/1/2/3
StrobeEnable 当前=True
TimerSelector 读取失败                  ← 无 Timer 节点，TimerActive 方案不可用
LineInverter=False                     ← 不反相，高电平有效 (配合 PNP)
```

| 输出方案 | 支持? | 原因 |
|----------|-------|------|
| `UserOutput` 软件直控电平 | ❌ | LineMode 固定为 Strobe，LineSource 不支持任何 UserOutput |
| `TimerActive` 定时器脉冲 | ❌ | TimerSelector 节点不存在 |
| `ExposureStartActive` 曝光频闪 | ✅ **唯一可用** | 每次相机曝光时 Line1 同步输出脉冲 |

#### 4.6.2 实际执行流程（`pulse_output_line` Attempt 2）

由于 `ExposureStartActive` 的脉冲只在曝光期间产生，必须**临时制造一次软触发曝光**：

```
① stop_grabbing()                          # 停流 — SDK 禁止在 streaming 时修改 TriggerSource
② LineSource = "ExposureStartActive"        # Line1 绑定到曝光开始事件
③ StrobeEnable = True                       # 开启频闪
④ StrobeLineDuration = 500ms (pulse_us)     # 脉冲宽度 (关键参数)
⑤ StrobeLineDelay = 0                       # 脉冲延迟 (部分相机不支持 StrobeLinePreDelay)
⑥ TriggerMode = On                          # 开启触发模式
⑦ TriggerSource = "Software"                # 临时切换到软触发
⑧ start_grabbing()                          # 启流 — 软触发需要 streaming 状态
⑨ TriggerSoftware()                         # 执行一次软触发 → 相机曝光 → Line1 输出脉冲
⑩ sleep(500ms + 0.2s)                       # 等待脉冲结束
⑪ stop_grabbing()                           # 停流 — 恢复外部触发模式前需要停流
⑫ set_external_trigger_mode()               # 恢复 Line0 外部触发模式
⑬ start_grabbing()                          # 重新进入 IO 监听待机状态
⑭ MV_CC_ClearImageBuffer()                  # ⚠️ 清除步骤⑨产生的脏帧
```

#### 4.6.3 关键注意事项

**a) 必须停流才能修改 TriggerSource**

海康 SDK 在 `grabbing` 状态下禁止修改 `TriggerSource`、`TriggerMode` 等关键节点。旧代码在相机 streaming 时直接调用 `SetEnumValueByString("TriggerSource", "Software")` 静默失败（未检查返回值），导致 `TriggerSoftware` 虽然返回 `MV_OK` 但因 TriggerSource 实际未切换而**无任何效果**——相机不曝光、Line1 无脉冲。

**b) 脏帧污染问题**

`TriggerSoftware` 命令导致相机曝光并产生一帧图像进入缓冲区。如果不调用 `MV_CC_ClearImageBuffer()`，下一轮 IO 监听周期的 `grab_frame()` 会立即拿到这帧"软件触发产生的垃圾图"，而不是等待 Line0 的真实触发。**已在 `finally` 块末尾添加 `ClearImageBuffer()` 解决。**

**c) StrobeLinePreDelay 节点不存在**

部分海康相机型号不支持 `StrobeLinePreDelay` 节点（返回 `0x80000106`）。该参数为非关键参数（设为 0），已改为不检查返回值，静默跳过。

**d) 所有 SDK 调用必须检查返回值**

修改前有 5 处 SDK 调用未检查返回值（`SetEnumValue` ×2、`SetIntValueEx` ×3），全部改为 `_raise_for_ret()` 抛出明确错误信息，杜绝静默失败。

#### 4.6.4 信号逻辑（PLC 视角）

```
检测结果 = OK →
  相机执行软触发曝光 → Line1 输出 500ms 高电平脉冲 (PNP 24V)
  → PLC DI 检测到脉冲 → 判定 OK

检测结果 = NG →
  相机不执行任何操作 → Line1 保持低电平 (0V)
  → PLC DI 超时未检测到脉冲 → 判定 NG
```

这是一种 **"只发 OK 脉冲"** 的单向信号机制。PLC 端通过超时判断识别 NG（当前超时值 3-5s，触发间隔 8s）。

#### 4.6.5 PNP 接线确认

```
海康相机 Line1+ (PNP输出) ────→ 三菱 PLC DI 输入端 (漏型/Sink)
        GND ──────────────────→ PLC COM
```

- PNP 输出：Line1 激活时输出 24V 高电平（电流从 Line1 流向 PLC 输入端）
- `LineInverter = False`：不反相，逻辑 1 = 高电平，与 PNP 接线一致
- 若需要改为 NPN（低电平有效），设置 `LineInverter = True` 即可

#### 4.6.7 `_capture_mode` 缓存导致间歇性失败（2026-06-12）

**现象**：Line1 OK 脉冲输出前两次正常，第三次及以后完全无电压变化。

**根因**：`CameraService` 维护了一个 `_capture_mode` 缓存变量，用于避免重复配置相机。但 `pulse_output_line` 执行过程中相机经历了一次完整的模式切换：

```
外部触发模式 → stop → Software触发 → start → 软触发曝光 → stop → 恢复外部触发 → start
```

`pulse_output_line` 返回后 `_capture_mode` **仍为** `"plc_external"`。前两次没出问题是因为相机内部状态恰好干净。但累积的 stop/start/模式切换 会逐渐污染相机内部状态（Line1 strobe 配置残留、AcquisitionMode 可能漂移、缓冲区碎片），第三次开始 `_prepare_capture_mode("plc_external")` 发现缓存命中直接 return，**跳过了重配**，让相机带着脏状态进入下一轮检测周期。

**修复**（`camera_service.py:emit_pass_output`）：

```python
try:
    client.pulse_output_line(line_name=output_channel, pulse_ms=pulse_ms, delay_ms=delay_ms)
except HikCameraError as exc:
    raise CameraServiceError(str(exc)) from exc
finally:
    # 脉冲输出过程中相机触发模式经历了临时切换，
    # 强制下一轮 capture_frame 完整重配相机，避免状态污染
    self._capture_mode = None
```

效果：每轮检测周期都从确定的初始状态出发，确保 `set_external_trigger_mode()` 完整执行。

#### 4.6.8 `configure_output_line` 取流中调用导致脉冲完全失效（2026-06-12）

**现象**：应用 4.6.7 修复后，Line1 脉冲输出完全失效——一次都无法产生电压变化。

**根因**：`pulse_output_line` 旧代码执行顺序存在致命缺陷：

```
configure_output_line()   ← 相机仍在 grabbing！LineSelector 设置可能被 SDK 静默忽略
_read_enum_node()          ← 探测 TimerSelector（不影响）
if timer_available: ...    ← Attempt 1 尝试
stop_grabbing()            ← 此时才停流，但 LineSelector 早已设过
configure strobe...        ← LineSource/StrobeEnable/StrobeLineDuration 可能配到了错误的线
```

海康 SDK 在取流状态下对部分节点的写入操作可能返回 `MV_OK` 但实际未生效。`LineSelector` 若静默失败，后续所有 `LineSource`、`StrobeEnable`、`StrobeLineDuration` 等配置全部作用到相机默认选中的线（可能是 Line0 或未定义状态），导致 Line1 完全无输出。

**修复**（`hik_camera_client.py:pulse_output_line`）：

```python
# ── 所有输出线配置必须在停流后进行 ──
was_grabbing = self._grabbing
if was_grabbing:
    self.stop_grabbing()          # ① 先停流

try:
    self.configure_output_line()   # ② 再选线（此时安全）
    # ... Attempt 1 / Attempt 2 ...
finally:
    self.set_external_trigger_mode()  # ③ 恢复外部触发
    if was_grabbing:
        self.start_grabbing()         # ④ 恢复取流
    self._camera.MV_CC_ClearImageBuffer()
```

同时新增**逐步骤日志**，每个 SDK 调用均记录返回值（16 进制），便于快速定位故障点：

```
pulse_output_line: stop grabbing before config (was_grabbing=True)
pulse_output_line: LineSelector=Line1 OK
pulse_output_line: TimerSelector available=False
pulse_output_line: falling back to Attempt 2 (ExposureStartActive strobe)
pulse_output_line: LineSource=ExposureStartActive OK
pulse_output_line: StrobeEnable=True OK
pulse_output_line: StrobeLineDuration=2000000 us OK
pulse_output_line: StrobeLineDelay=0 us OK
pulse_output_line: StrobeLinePreDelay=0 ret=0x80000106 (non-critical)
pulse_output_line: TriggerMode=On OK
pulse_output_line: TriggerSource=Software OK
pulse_output_line: start_grabbing OK, firing TriggerSoftware...
pulse_output_line: TriggerSoftware OK, sleeping 2.200 s
pulse_output_line: stop_grabbing OK, restoring external trigger mode
pulse_output_line: external trigger mode restored
pulse_output_line: grabbing restarted
pulse_output_line: ClearImageBuffer done, pulse sequence complete
```

#### 4.6.10 `StrobeEnable` 按线分隔——全流程 Line1 持续高电平（2026-06-12）

**现象**：全流程检测 OK → Line1 脉冲输出后电平不回落，一直保持高电平。但设置对话框的「手动测试 Line1 OK 脉冲」功能正常。

**根因**：海康相机上 **`StrobeEnable`、`LineSource`、`StrobeLineDuration` 都是按 `LineSelector` 分线存储** 的参数。即修改 `StrobeEnable` 只影响当前 `LineSelector` 选中的线。

`set_external_trigger_mode()`（外部触发模式恢复时）和 `pulse_output_line()`（脉冲结束后）都调用了 `StrobeEnable=False`，但**没有先选中 Line1**。此时 `StrobeEnable=False` 被写到了当前 `LineSelector` 指向的线（可能是 Line0 或其他线），**Line1 的频闪未被关闭**。

结果：相机回到外部触发待机模式后，每次真实 Line0 触发曝光，Line1 都跟着输出 `ExposureStartActive` 频闪脉冲（脉冲宽度 = 上次设定的 `StrobeLineDuration` ⊘ 2 秒），PLC 端持续的 24V 高电平被解读为"一直高"。

**修复**（`hik_camera_client.py`）：

新增 `_disable_strobe_on_line(line_name, context)` 辅助函数，三步确保关闭：
```python
def _disable_strobe_on_line(self, line_name: str, context: str = "") -> None:
    # ① 先选中目标输出线
    ret_sel = self._camera.MV_CC_SetEnumValueByString("LineSelector", line_name)
    if ret_sel != MV_OK: ...
    # ② 关闭该线的频闪
    ret_sd = self._camera.MV_CC_SetBoolValue("StrobeEnable", False)
    if ret_sd != MV_OK: ...
    # ③ 回读确认
    readback = self._read_bool_node("StrobeEnable")
    _logger.info("%s%s StrobeEnable 回读=%s (期望 False)", prefix, line_name, readback)
```

`set_external_trigger_mode` 和 `pulse_output_line` 收尾处均改为调用此函数。日志回读确认 `StrobeEnable 回读=False`。

**为什么手动测试正常但全流程异常**：手动测试时 `was_grabbing=False`（未启动 IO 监听），`pulse_output_line` 内部 `configure_output_line("Line1")` 先选中了 Line1，所以后续 `StrobeEnable=False` 正确作用于 Line1。全流程时 `set_external_trigger_mode` 被 `_prepare_capture_mode` 重复调用，且未先选线，导致 Line1 频闪未被关闭。

#### 4.6.11 手动模式缺少脉冲输出（2026-06-12）

**现象**：主界面「手动测试」按钮 → 检测结果 OK，但 Line1 无脉冲输出。设置对话框「手动测试 Line1 OK 脉冲」正常。

**根因**：`InspectionWorkflowService.execute_inspection()` 中只对 `trigger_source in {"plc", "io"}` 调用 `_handle_plc_result()`。主界面「手动测试」走的是 `trigger_source="manual"`，不在此集合内，直接跳过了 PLC 结果处理——即使检测结果为 OK 也不会输出 Line1 脉冲。

**三种触发路径对比**：

| 触发方式 | trigger_source | OK 时输出脉冲？ |
|----------|---------------|-----------------|
| 主界面「手动测试」按钮 | `manual` | ~~❌~~ → ✅ (修复后) |
| 设置 →「手动测试 Line1 OK 脉冲」 | （直接调 `emit_pass_output`） | ✅ |
| IO 监听 + Line0 触发 | `io` | ✅ |
| 「PLC 触发」按钮 | `plc` | ✅ |

**修复**（`inspection_workflow_service.py:107`）：

```python
# 修改前
if trigger_source in {"plc", "io"}:

# 修改后
if trigger_source in {"plc", "io", "manual"}:
```

注意：`_handle_plc_result()` 内部仍有 `plc_config.enabled` 和 `trigger_mode == "plc_external"` 双重校验，只有配方启用 PLC 且配置为外部触发模式时，OK 结果才会输出脉冲。手动采图只是多了这条路，不会改变 NG 的处理逻辑（NG 始终不输出）。

## 六、换型功能

支持两种换型模式，可在设置中配置，也可在主界面工具栏快速切换：

### 6.1 自动换型（默认）

- 流水号有效性校验：非空、非 `"0"`、长度恰好为 10，否则丢弃等待下次
- 每次 Line0 触发或手动输入流水号时，PC 通过 **HTTP API** (`POST /api/v1/crankshaft/model-by-serial`) 查询机型，不再直接查 SQL Server
- API 返回完整机型号（如 `10V3AABB1234`），取**前 4 位**作为配方键（`10V3`）匹配本地配方文件
- 幂等去重：同一流水号仅处理一次，避免重复换型
- 若机型与当前不同 → 自动加载新配方 → 主界面状态栏显示换型通知（"自动换型：163 → 208"）
- 机型显示区为**只读**，操作员无法手动修改
- `app_config.json` 字段：`"switch_mode": "auto"`，API 地址在 `crankshaft_api.base_url` 配置

### 6.2 手动换型

- 软件**忽略** SQL Server 的机型字段，不发起查询
- 主界面右侧显示机型 **下拉选择框**（列出所有已配置配方）+ "确认切换"按钮
- 切换时弹出 `QMessageBox` 二次确认，防止误操作
- 选定后配方**锁定**，直到操作员再次手动切换
- `app_config.json` 字段：`"switch_mode": "manual"`

## 七、检测算法

每个机型的配方独立配置检测算法，两种算法可按机型混用：

### 7.1 传统二值化（默认）

1. 对配置的 ROI 区域裁剪图像
2. **灰度二值化**（三种模式可选）：
   - `otsu`：Otsu 自适应阈值，自动计算最优阈值
   - `fixed`：固定阈值，`gray > threshold_value` 的像素为白色
   - `range`：灰度区间，`gray_low ≤ gray ≤ gray_high` 的像素为白色
3. **目标明暗（polarity）**：
   - `bright`：目标为亮区（白色）不变
   - `dark`：目标为暗区，自动反转二值图
4. **手动反转（invert）**：可手动反转二值结果
5. 形态学开运算去噪（morph_kernel 可调）
6. `cv2.connectedComponentsWithStats` 统计连通域面积
7. **面积判定**：
   - `max_area = 0`：`detected_area ≥ min_area` 即合格
   - `max_area > 0`：`min_area ≤ detected_area ≤ max_area` 才合格（排除过大噪点/反光）

### 7.2 AI 模型（YOLOv8）

1. 加载 `.pt` 文件（`ultralytics.YOLO`）
2. 对 ROI 区域推理
3. 按类别 + 置信度阈值判断
4. 不含训练/标注工具，仅推理

## 十三、相机参数手动控制

### 13.1 功能目标

在主界面提供相机基础参数的实时读取/设置面板，替代海康 MVS 客户端的手动调参流程。现场调试时，操作员无需切换到 MVS 即可完成曝光时间、增益等常用参数的调整。

### 13.2 可控参数

通过 GenICam 标准节点名访问相机参数，按相机型号自动探测可用参数：

| 参数 | 节点名 | 类型 | 典型范围 | 说明 |
|------|--------|------|----------|------|
| 曝光时间 | `ExposureTime` | Float | 20 ~ 1,000,000 μs | 影响图像亮度，值越大图像越亮 |
| 模拟增益 | `GainRaw` | Float/Int | 0 ~ 24 dB | 信号放大倍数，过高引入噪点 |
| Gamma | `Gamma` | Float | 0.1 ~ 4.0 | 灰度非线性变换 |
| 帧率 | `AcquisitionFrameRate` | Float | 1 ~ 相机最大帧率 | 采集帧率控制 |
| 数字增益 | `DigitalGain` | Float | 1.0 ~ 16.0 | 数字域增益，部分相机不支持 |

> **注意**：实际可用参数取决于相机型号。`HikCameraClient` 通过 `MV_CC_GetFloatValue` / `MV_CC_SetFloatValue` 通用接口读写，对不支持的参数静默跳过并记录 WARN 日志。

### 13.3 UI 位置与布局

主界面右侧面板，在「检测结果」与「统计信息」之间新增 `QGroupBox("相机参数")`：

```
┌─ 相机参数 ──────────────────────┐
│  曝光时间：[  5000] μs  ↕      │
│  增益：    [   0.0] dB  ↕      │
│  Gamma：   [   1.0]     ↕      │
│  帧率：    [  30.0] fps ↕      │
│                                 │
│  [读取当前值]    [应用]         │
└─────────────────────────────────┘
```

### 13.4 交互方式

- **参数控件**：`QDoubleSpinBox`，范围和步长根据参数类型设定
  - 曝光时间：20 ~ 1,000,000 μs，步长 100
  - 增益：0 ~ 24 dB，步长 0.1
  - Gamma：0.1 ~ 4.0，步长 0.1
  - 帧率：1 ~ 120 fps，步长 1
- **读取当前值**：调用 `camera_controller.get_camera_params()` 刷新所有 SpinBox 显示
- **应用**：将当前面板值逐个通过 `camera_controller.set_camera_param()` 下发到相机
- **自动应用**（可选优化）：SpinBox 值改变后 300ms debounce 自动下发，减少手动点击
- **软件启动时**：自动从 `app_config.json` 恢复上次参数，并尝试写入相机

### 13.5 联动锁定逻辑

- **IO 监听运行中**：所有相机参数控件置灰（`setEnabled(False)`），防止运行时修改影响在线检测
- **相机未连接**：参数控件置灰，面板显示"相机未连接"
- **参数读取失败**：对应 SpinBox 保留上次值，状态栏提示读取失败信息

### 13.6 持久化

在 `app_config.json` 的 `camera` 节点中存储最近一次设置的参数值（详见第九章），软件启动时恢复到相机，软件正常退出时保存当前值。

## 十五、验证步骤

1. **相机测试**：无 PLC 时改用软件触发（`TriggerSource=Software`），验证采图正常
2. **数据库测试**：设置界面点击"测试连接"，验证 `SELECT MachineType FROM T_SerialNo WHERE ActSerialNo='xxx'` 返回正确
3. **IO 测试**：用万用表测 Line1，合格时测到高电平持续 ~500ms 后回低
4. **IO 联调**（详见第四章完整清单）：
   - 接线确认：PLC ↔ 相机 Line0/Line1
   - 相机节点确认：TriggerMode / TriggerSource / TriggerActivation / LineSelector
   - 软件侧检查：配方 trigger_mode / IO 监听启动 / 触发响应
   - 结果回传验证：OK/NG 样本测试、示波器脉宽测量
5. **自动换型测试**：启动上位机 HTTP 服务，输入有效流水号（10位非0），验证软件通过 API 获取完整机型号并提取前4位匹配配方（如返回 `10V3AABB1234` → 加载 `10V3.json`）；输入无效流水号验证被状态栏提示拒绝
6. **手动换型测试**：切换至手动模式，选择机型，确认换型锁定，再次 Line0 触发不改变机型
7. **二值化调试**：
   - 拍摄或加载图片 → 鼠标拖拽框选 ROI → 表格同步更新
   - 切换阈值方式（Otsu / 固定 / 灰度区间）
   - 调节目标明暗（bright / dark）+ 反转
   - 设置面积上下限（min_area / max_area）
   - 右侧面板实时预览：原图 | Otsu二值 | 形态学开运算 | 连通域标注

### v1.10 (2026-06-17) — stop_grabbing CALLORDER 状态不同步修复

**问题**：手动采图偶发 `capture 失败: stop grabbing failed: 0x80000003`。错误码 `0x80000003` = `MV_E_CALLORDER`（函数调用顺序错误），SDK 认为相机当前不在取流状态，拒绝执行 `StopGrabbing`。

**根因**：`HikCameraClient` 内部的 `_grabbing` (Python 标志位) 与 SDK 实际相机状态**不同步**。当 Python 认为相机正在取流但 SDK 实际已停止取流时，任何调用 `stop_grabbing()` 的地方都会触发 CALLORDER 错误，且由于 `_grabbing` 在异常抛出前未更新，导致**永久卡死**——后续所有 capture 操作都会在同一位置失败。

**状态不同步的典型触发场景**：`pulse_output_line()` (Line1 OK 脉冲输出) 内部执行复杂的 start/stop grabbing 序列（Attempt 1~5 共 5 级回退尝试）。如果在某个 attempt 内 `start_grabbing()` 成功后、`stop_grabbing()` 之前发生异常（SDK 错误、网络抖动等），`_grabbing` 会保持 `True` 不更新，但 SDK 实际状态已不可控。后续 `_prepare_capture_mode()` 检测到 `is_grabbing=True` 再次调用 `stop_grabbing()` 时触发 CALLORDER。

**修复** (`infrastructure/camera/hik_camera_client.py`)：

1. **`stop_grabbing()`** — 新增 `MV_E_CALLORDER` 优雅处理：当 SDK 返回 CALLORDER 时，说明相机当前未在取流，将 `_grabbing` 同步为 `False` 并记录 WARN 日志即可，**不再抛异常**。这既自动修复了状态不同步，又避免了永久卡死。

```python
def stop_grabbing(self) -> None:
    self._ensure_opened()
    if not self._grabbing:
        return
    ret = self._camera.MV_CC_StopGrabbing()
    if ret == MV_E_CALLORDER:
        # CALLORDER on StopGrabbing → SDK believes we are not streaming.
        # Sync the flag and move on — forcing an error here would leave
        # the flag stuck at True permanently.
        _logger.warning("stop_grabbing: SDK returned CALLORDER (not grabbing); syncing flag to False")
        self._grabbing = False
        return
    self._raise_for_ret("stop grabbing", ret)
    self._grabbing = False
```

2. **`start_grabbing()`** — 同样对 `MV_E_CALLORDER` 做防护：SDK 返回 CALLORDER 通常表示相机已在取流中，将 `_grabbing` 同步为 `True` 并继续。

3. **新增 import**：从 `MvErrorDefine_const` 导入 `MV_E_CALLORDER`。

**效果**：即使 `_grabbing` 标志位因异常漂移，下一次 capture 操作也能自动恢复，不会再陷入"永久失败"状态。
